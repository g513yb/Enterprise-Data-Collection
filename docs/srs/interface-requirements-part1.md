## 1. 概述

本文档定义**云南省企业就业失业数据采集系统**的数据采集接口需求，是功能需求说明书的配套接口设计文档。

系统涉及三级用户：**企业（FR-1xx）**、**市局（FR-3xx）**、**省局（FR-2xx）**，并需与**国家失业监测系统**进行数据对接。本文件（前半部分）覆盖企业端与市级审核的所有数据采集接口。

**前半部分接口范围：**

| 编号 | 接口名称 | 方向 | 对应功能需求 |
| :--- | :--- | :--- | :--- |
| IF-01 | 企业备案上报 | 企业 → 省局 | FR-102 |
| IF-02 | 备案审核结果查询 | 企业 → 省局 | FR-102 |
| IF-03 | 就业数据草稿保存 | 企业 → 省局 | FR-103 |
| IF-04 | 就业数据正式上报 | 企业 → 省局（经市局） | FR-103 |
| IF-05 | 企业历史数据查询 | 企业 → 省局 | FR-104 |
| IF-06 | 市级数据审核 | 市局 → 省局 | FR-301 |
| IF-07 | 市级数据退回 | 市局 → 企业 | FR-301 |

> **后半部分**将覆盖：省局数据汇总上报（IF-08）、省局数据退回（IF-09）、通知发布（IF-10）、系统管理（IF-11）等接口。

## 2. 术语说明

| 术语 | 说明 |
| :--- | :--- |
| 调查期 | 省局配置的数据填报窗口期，格式为年月（如 `202403`）或年季（如 `2024Q1`） |
| 上报期 | 同"调查期"，指企业必须在此期间内完成数据填报的时间范围 |
| 备案 | 企业将基础信息提交至省局，经省局审核通过后方可参与后续填报的流程 |
| 组织机构代码 | 企业统一编码，仅含字母和数字，不超过9位 |
| 建档期就业人数 | 企业初次建档时填写的监测点就业人数 |
| 调查期就业人数 | 本次调查期内企业监测点当时的就业人数 |
| 就业人数减少类型 | 当调查期就业人数低于建档期时，必须填写的减少原因分类 |
| 数据状态 | 数据在系统工作流中的当前状态（见各接口状态说明） |

## 3. 通用接口规范

### 3.1 数据交换格式

所有接口采用 **XML** 格式进行数据交换，统一使用以下请求信封：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Request>
  <Header>
    <SystemID>YNYJSJ</SystemID>           <!-- 云南省就业失业数据采集系统 -->
    <Version>1.0</Version>
    <Timestamp>2024-03-01T09:00:00</Timestamp>
    <RequestID>REQ20240301090000001</RequestID>
  </Header>
  <Body>
    <!-- 各接口业务数据 -->
  </Body>
</Request>
```

所有接口返回统一响应信封：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Response>
  <Header>
    <RequestID>REQ20240301090000001</RequestID>
    <Timestamp>2024-03-01T09:00:05</Timestamp>
  </Header>
  <Result>
    <Code>0000</Code>
    <Message>操作成功</Message>
  </Result>
  <Body>
    <!-- 各接口返回数据 -->
  </Body>
</Response>
```

### 3.2 字符编码

| 属性 | 值 |
| :--- | :--- |
| 编码 | UTF-8 |
| 语言 | 简体中文（zh-CN） |
| 换行符 | CRLF（`\r\n`） |

---

### 3.3 日期与时间格式

| 类型 | 格式 | 示例 |
| :--- | :--- | :--- |
| 日期 | `YYYY-MM-DD` | `2024-03-01` |
| 日期时间 | `YYYY-MM-DDTHH:mm:ss` | `2024-03-01T09:00:00` |
| 调查期（月度） | `YYYYMM` | `202403` |
| 调查期（季度） | `YYYYQ#` | `2024Q1` |

---

### 3.4 错误码说明

| 错误码 | 含义 | 说明 |
| :--- | :--- | :--- |
| `0000` | 成功 | 请求处理成功 |
| `1001` | 参数错误 | 必填字段缺失或格式不符合要求 |
| `1002` | 身份验证失败 | 系统标识或凭证无法通过验证 |
| `1003` | 重复提交 | 相同主键的记录已存在 |
| `1004` | 不在上报期内 | 当前时间不在任何开放的调查期窗口内 |
| `1005` | 备案未通过 | 企业尚未完成备案审核，不可进行数据填报 |
| `2001` | 记录不存在 | 查询的记录不存在 |
| `2002` | 状态冲突 | 当前记录的状态不允许执行此操作 |
| `2003` | 权限不足 | 当前用户无权执行此操作 |
| `9999` | 系统异常 | 服务端发生未知错误 |

## 5. 企业端接口

---

### 5.1 IF-01 企业备案上报接口

**接口编号：** `IF-01`  
**对应功能需求：** FR-102 备案上报  
**调用方向：** 企业系统 → 省局系统  
**触发时机：** 企业用户完成基础信息填写后点击"上报备案"  
**前置条件：** 省级管理员已创建企业账号  
**后置条件：** 备案记录创建，状态为`待审核`

#### 请求字段说明

| 字段名 | XML标签 | 类型 | 必填 | 校验规则 |
| :--- | :--- | :--- | :--- | :--- |
| 企业账号ID | `<EnterpriseID>` | 字符串 | 是 | 系统分配，不可修改 |
| 组织机构代码 | `<OrgCode>` | 字符串 | 是 | 仅字母和数字，不超过9位 |
| 企业名称 | `<EntName>` | 字符串 | 是 | 中文或英文，不超过100字 |
| 企业性质（一级） | `<EntTypeL1>` | 字符串 | 是 | 受控词汇，两级下拉 |
| 企业性质（二级） | `<EntTypeL2>` | 字符串 | 是 | 受控词汇，从属于一级 |
| 所属行业（一级） | `<IndustryL1>` | 字符串 | 是 | 受控词汇，两级下拉 |
| 所属行业（二级） | `<IndustryL2>` | 字符串 | 是 | 受控词汇，从属于一级 |
| 主要经营业务 | `<MainBusiness>` | 字符串 | 是 | 不超过500字 |
| 所属地市 | `<RegionCity>` | 字符串 | 是 | 云南省地区编码表，不可修改 |
| 所属市县 | `<RegionCounty>` | 字符串 | 是 | 云南省地区编码表，不可修改 |
| 所处区域 | `<RegionDistrict>` | 字符串 | 是 | 云南省地区编码表，不可修改 |
| 联系人 | `<ContactName>` | 字符串 | 是 | 中文或英文，不超过50字 |
| 联系地址 | `<ContactAddr>` | 字符串 | 是 | 两级下拉选择 |
| 邮政编码 | `<PostalCode>` | 字符串 | 是 | 必须为6位数字 |
| 联系电话 | `<Phone>` | 字符串 | 是 | 格式：（区号）+电话号码 或 手机号码 |
| 传真 | `<Fax>` | 字符串 | 是 | 格式：（区号）+电话号码 |
| 电子邮箱 | `<Email>` | 字符串 | 否 | 格式须符合 `xxx@xxx.xxx` |

#### 请求示例

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Request>
  <Header>
    <SystemID>YNYJSJ</SystemID>
    <Version>1.0</Version>
    <Timestamp>2024-03-01T09:00:00</Timestamp>
    <RequestID>REQ20240301090000001</RequestID>
  </Header>
  <Body>
    <Interface>IF-01</Interface>
    <EnterpriseID>ENT000123</EnterpriseID>
    <OrgCode>A12345678</OrgCode>
    <EntName>云南示例科技有限公司</EntName>
    <EntTypeL1>有限责任公司</EntTypeL1>
    <EntTypeL2>一人有限责任公司</EntTypeL2>
    <IndustryL1>制造业</IndustryL1>
    <IndustryL2>电子设备制造</IndustryL2>
    <MainBusiness>电子元器件研发与制造</MainBusiness>
    <RegionCity>530100</RegionCity>
    <RegionCounty>530102</RegionCounty>
    <RegionDistrict>530102001</RegionDistrict>
    <ContactName>张三</ContactName>
    <ContactAddr>昆明市五华区人民路1号</ContactAddr>
    <PostalCode>650000</PostalCode>
    <Phone>0871-12345678</Phone>
    <Fax>0871-12345679</Fax>
    <Email>zhangsan@example.com</Email>
  </Body>
</Request>
```

#### 返回字段说明

| 字段名 | XML标签 | 类型 | 说明 |
| :--- | :--- | :--- | :--- |
| 结果码 | `<Code>` | 字符串 | `0000` 表示成功 |
| 结果描述 | `<Message>` | 字符串 | 操作结果说明 |
| 备案记录ID | `<FilingID>` | 字符串 | 系统生成的备案唯一标识 |
| 备案状态 | `<Status>` | 字符串 | `待审核` |
| 提交时间 | `<SubmitTime>` | 日期时间 | 服务端记录的提交时间 |

#### 返回示例

```xml
<Response>
  <Header>
    <RequestID>REQ20240301090000001</RequestID>
    <Timestamp>2024-03-01T09:00:02</Timestamp>
  </Header>
  <Result>
    <Code>0000</Code>
    <Message>备案信息上报成功，等待省局审核。</Message>
  </Result>
  <Body>
    <FilingID>FIL20240301000001</FilingID>
    <Status>待审核</Status>
    <SubmitTime>2024-03-01T09:00:02</SubmitTime>
  </Body>
</Response>
```

#### 业务规则

- 每个企业同一时间只能有**一条有效备案记录**。仅当上一次备案状态为`已退回`时，方可重新提交。
- `RegionCity`、`RegionCounty`、`RegionDistrict` 须与账号绑定地区一致，企业不可自行修改，系统自动校验。
- `OrgCode` 在全系统范围内须唯一，重复提交返回错误码 `1003`。
- `Email` 非必填，但若填写则必须通过格式校验，否则返回 `1001`。

### 5.2 IF-02 备案审核结果查询接口

**接口编号：** `IF-02`  
**对应功能需求：** FR-102 备案上报  
**调用方向：** 企业系统 → 省局系统  
**触发时机：** 企业用户查看当前备案状态  
**前置条件：** 企业已提交过备案信息  
**后置条件：** 返回当前备案状态及退回原因（如有）

#### 请求字段说明

| 字段名 | XML标签 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 企业账号ID | `<EnterpriseID>` | 字符串 | 是 | |
| 备案记录ID | `<FilingID>` | 字符串 | 否 | 不填则返回最新备案记录 |

#### 返回字段说明

| 字段名 | XML标签 | 类型 | 说明 |
| :--- | :--- | :--- | :--- |
| 备案记录ID | `<FilingID>` | 字符串 | |
| 备案状态 | `<Status>` | 字符串 | 见下方状态说明 |
| 审核人 | `<Reviewer>` | 字符串 | 省局审核人员姓名，审核后填写 |
| 审核时间 | `<ReviewTime>` | 日期时间 | 审核后填写 |
| 退回原因 | `<ReturnRemark>` | 字符串 | 状态为`已退回`时必有值 |

#### 备案状态值说明

| 状态值 | 说明 |
| :--- | :--- |
| `待审核` | 企业已提交，等待省局审核 |
| `已通过` | 省局审核通过，企业可进行数据填报 |
| `已退回` | 省局审核退回，企业可查看退回原因并重新提交 |

### 5.3 IF-03 就业数据草稿保存接口

**接口编号：** `IF-03`  
**对应功能需求：** FR-103 数据填报（保存草稿功能）  
**调用方向：** 企业系统 → 省局系统  
**触发时机：** 企业用户填报过程中点击"保存草稿"  
**前置条件：**
- 企业备案状态为`已通过`
- 当前时间在开放的调查期内

**后置条件：** 草稿记录保存，状态为`草稿`，不触发审核流程

#### 请求字段说明

| 字段名 | XML标签 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 企业账号ID | `<EnterpriseID>` | 字符串 | 是 | |
| 调查期 | `<SurveyPeriod>` | 字符串 | 是 | 格式：`YYYYMM` |
| 建档期就业人数 | `<FilingEmployment>` | 整数 | 否 | 草稿允许不完整 |
| 调查期就业人数 | `<SurveyEmployment>` | 整数 | 否 | 草稿允许不完整 |
| 其他原因 | `<OtherReasons>` | 字符串 | 否 | |
| 就业人数减少类型 | `<DecreaseType>` | 字符串 | 否 | 草稿阶段不做强制校验 |
| 主要原因 | `<PrimaryReason>` | 字符串 | 否 | |
| 主要原因说明 | `<PrimaryReasonDesc>` | 字符串 | 否 | |
| 次要原因 | `<SecondaryReason>` | 字符串 | 否 | |
| 次要原因说明 | `<SecondaryReasonDesc>` | 字符串 | 否 | |
| 第三原因 | `<ThirdReason>` | 字符串 | 否 | |
| 第三原因说明 | `<ThirdReasonDesc>` | 字符串 | 否 | |

#### 返回示例

```xml
<Response>
  <Result>
    <Code>0000</Code>
    <Message>草稿保存成功。</Message>
  </Result>
  <Body>
    <DraftID>DFT202403150001</DraftID>
    <Status>草稿</Status>
    <SaveTime>2024-03-15T10:00:00</SaveTime>
  </Body>
</Response>
```

#### 业务规则

- 草稿阶段**不进行完整性校验**，允许字段部分为空。
- 同一企业同一调查期只能有一条草稿记录。重复保存时，以最新内容覆盖。
- 草稿**不触发**市级/省级审核流程。
- 调查期关闭后，草稿记录将被系统锁定，不可再修改或提交。

### 5.4 IF-04 就业数据正式上报接口

**接口编号：** `IF-04`  
**对应功能需求：** FR-103 数据填报（提交上报功能）  
**调用方向：** 企业系统 → 省局系统（经市局审核流转）  
**触发时机：** 企业用户在上报期内点击"提交上报"  
**前置条件：**
- 企业备案状态为`已通过`
- 当前时间在开放的调查期内
- 填报数据通过完整性校验

**后置条件：** 提交记录创建，状态为`已上报（待市级审核）`

#### 请求字段说明

| 字段名 | XML标签 | 类型 | 必填 | 校验规则 |
| :--- | :--- | :--- | :--- | :--- |
| 企业账号ID | `<EnterpriseID>` | 字符串 | 是 | |
| 调查期 | `<SurveyPeriod>` | 字符串 | 是 | 格式：`YYYYMM`，须为当前开放调查期 |
| 建档期就业人数 | `<FilingEmployment>` | 整数 | 是 | ≥ 0 |
| 调查期就业人数 | `<SurveyEmployment>` | 整数 | 是 | ≥ 0 |
| 其他原因 | `<OtherReasons>` | 字符串 | 是 | |
| 就业人数减少类型 | `<DecreaseType>` | 字符串 | 条件必填 | 当 `SurveyEmployment` < `FilingEmployment` 时必填，见受控值 |
| 主要原因 | `<PrimaryReason>` | 字符串 | 条件必填 | 当 `SurveyEmployment` < `FilingEmployment` 时必填，见受控值 |
| 主要原因说明 | `<PrimaryReasonDesc>` | 字符串 | 条件必填 | 填写主要原因时必填 |
| 次要原因 | `<SecondaryReason>` | 字符串 | 否 | 见受控值 |
| 次要原因说明 | `<SecondaryReasonDesc>` | 字符串 | 否 | |
| 第三原因 | `<ThirdReason>` | 字符串 | 否 | 见受控值 |
| 第三原因说明 | `<ThirdReasonDesc>` | 字符串 | 否 | |

#### 就业人数减少类型受控值（`<DecreaseType>`）

| 编码 | 中文名称 |
| :--- | :--- |
| `DT01` | 关闭破产 |
| `DT02` | 停业整顿 |
| `DT03` | 经济性裁员 |
| `DT04` | 业务转移 |
| `DT05` | 自然减员 |

#### 业务规则

- 仅当 `SurveyPeriod` 对应的调查期处于**开放状态**时，接口方接受请求；否则返回错误码 `1004`。
- 当 `SurveyEmployment` < `FilingEmployment` 时，`DecreaseType`、`PrimaryReason`、`PrimaryReasonDesc` 三个字段**强制必填**，否则返回 `1001`。
- 同一企业同一调查期只能提交**一条正式记录**。如该调查期已有记录且状态不为`已退回`，则返回 `1003`。
- 三个原因字段（`PrimaryReason`、`SecondaryReason`、`ThirdReason`）如同时填写，**编码不得重复**。
- 系统在接收到正式上报后，自动将对应草稿记录（如有）标记为`已提交`，不可再修改。

#### 请求字段说明

| 字段名 | XML标签 | 类型 | 必填 | 说明 |

| 字段名 | XML标签 | 类型 | 说明 |
| :--- | :--- | :--- | :--- |

### 6.1 IF-06 市级数据审核接口

**接口编号：** `IF-06`  
**对应功能需求：** FR-301 数据审核  
**调用方向：** 市局系统 → 省局系统  
**触发时机：** 市级用户完成审核操作（通过或退回）  
**前置条件：** 企业提交记录状态为`已上报（待市级审核）`  
**后置条件：**
- 审核通过：状态更新为`已上报（待省级审核）`
- 退回：状态更新为`已退回`，退回原因对企业可见

#### 请求字段说明

| 字段名 | XML标签 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 市局账号ID | `<MuniID>` | 字符串 | 是 | 系统分配的市级用户标识 |
| 提交记录ID | `<SubmissionID>` | 字符串 | 是 | 目标企业提交记录 |
| 审核操作 | `<Action>` | 字符串 | 是 | `APPROVE`（通过）或 `RETURN`（退回） |
| 退回原因 | `<ReturnRemark>` | 字符串 | 条件必填 | `Action` 为 `RETURN` 时必填，不超过500字 |
| 审核人ID | `<ReviewerID>` | 字符串 | 是 | 市级操作人用户ID，系统自动记录 |
| 审核时间 | `<ReviewTime>` | 日期时间 | 是 | 系统自动记录 |
| 是否批量审核 | `<IsBatch>` | 布尔值 | 否 | `true` 表示批量操作，默认 `false` |
| 批量记录ID列表 | `<SubmissionIDs>` | 列表 | 条件必填 | `IsBatch` 为 `true` 时必填 |

#### 请求示例（单条审核通过）

```xml
<Request>
  <Header>
    <SystemID>YNYJSJ</SystemID>
    <Version>1.0</Version>
    <Timestamp>2024-03-18T14:00:00</Timestamp>
    <RequestID>REQ20240318140000001</RequestID>
  </Header>
  <Body>
    <Interface>IF-06</Interface>
    <MuniID>MUNI530100</MuniID>
    <SubmissionID>SUB202403150001</SubmissionID>
    <Action>APPROVE</Action>
    <ReviewerID>USR_MUNI_001</ReviewerID>
    <ReviewTime>2024-03-18T14:00:00</ReviewTime>
    <IsBatch>false</IsBatch>
  </Body>
</Request>
```

#### 返回示例（审核通过）

```xml
<Response>
  <Result>
    <Code>0000</Code>
    <Message>审核通过，数据已转交省局审核。</Message>
  </Result>
  <Body>
    <SubmissionID>SUB202403150001</SubmissionID>
    <Status>已上报（待省级审核）</Status>
    <ReviewTime>2024-03-18T14:00:00</ReviewTime>
  </Body>
</Response>
```

#### 业务规则

- 市级用户**只能审核本辖区内**企业提交的数据，跨辖区操作返回错误码 `2003`。
- `Action` 为 `RETURN` 时，`ReturnRemark` 为**必填**，不得为空，否则返回 `1001`。
- 市级用户**不可修改**企业提交数据的内容，仅可进行通过或退回操作。
- 支持**批量审核**：`IsBatch` 为 `true` 时，`SubmissionIDs` 替代 `SubmissionID` 作为操作目标，批量操作需全部成功或全部回滚。
- 审核操作需记录：**审核人、审核时间、审核结果**，供后续查看审核记录（FR-301）使用。
- 已退回的记录，企业重新提交后，状态重置为`已上报（待市级审核）`，市级须重新审核。

### 6.2 IF-07 市级数据退回接口

**接口编号：** `IF-07`  
**对应功能需求：** FR-301 数据审核（退回功能）  
**调用方向：** 市局系统 → 企业（状态通知）  
**触发时机：** 市级用户点击"退回"并确认退回原因  
**前置条件：** 提交记录状态为`已上报（待市级审核）`或`市级审核中`  
**后置条件：**
- 数据状态更新为`已退回`
- 企业端可查看退回原因，并在调查期内重新填报

#### 请求字段说明

| 字段名 | XML标签 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 市局账号ID | `<MuniID>` | 字符串 | 是 | |
| 提交记录ID | `<SubmissionID>` | 字符串 | 是 | |
| 退回原因 | `<ReturnReason>` | 字符串 | 是 | 不超过500字，企业端可见 |
| 退回人ID | `<ReturnUserID>` | 字符串 | 是 | 市级操作人用户ID |
| 退回时间 | `<ReturnTime>` | 日期时间 | 是 | 系统自动记录 |
| 批量退回ID列表 | `<SubmissionIDs>` | 列表 | 否 | 批量退回时使用 |

#### 返回示例

```xml
<Response>
  <Result>
    <Code>0000</Code>
    <Message>数据已退回，企业端已收到退回通知。</Message>
  </Result>
  <Body>
    <SubmissionID>SUB202403150001</SubmissionID>
    <Status>已退回</Status>
    <ReturnTime>2024-03-18T15:00:00</ReturnTime>
  </Body>
</Response>
```

#### 退回日志记录字段

系统自动记录每次退回操作，日志字段如下：

| 字段 | 说明 |
| :--- | :--- |
| 退回人 | 当前操作的市级用户账号 |
| 退回时间 | 系统自动记录 |
| 退回原因 | 市级用户填写内容，企业端可见 |
| 退回数据 | 被退回记录的 `SubmissionID` 列表 |

#### 业务规则

- 退回后，企业可在**当前调查期仍开放**的前提下重新填报并提交。
- 若调查期已关闭，退回记录将被锁定，企业无法再次提交该期数据。
- 退回原因对**企业用户可见**，系统在企业端的数据查询（FR-104）中展示。
- 支持批量退回，使用 `SubmissionIDs` 字段传入多条记录ID，操作结果逐条返回。

# 4 非功能需求

## 4.1 性能需求 

系统需确保在全省范围内的高频访问下具备良好的响应能力。

| ID | 需求名称 | 需求描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **NFR-101** | **并发负载能力** | 系统必须能够承载全省各市、县、企业级用户在上报期内的集中式高频访问压力。 | 在模拟 500 个用户同时执行“数据提交”操作的压力测试下，系统失败率需低于 0.1%，且 CPU 占用率平稳保持在 75% 以下。 |
| **NFR-102** | **交互响应效率** | 针对企业端的基础业务操作（如登录验证、备案信息修改等），系统需提供即时的反馈体验。 | 95% 以上的常规页面请求（HTTP Get/Post）端到端响应时间不得超过 2 秒。 |
| **NFR-103** | **大数据处理性能** | 针对省级用户发起的全省范围就业数据汇总、多维趋势分析及导出任务。 | 涉及全省数万条记录的对比分析报表及图形化统计（饼图、折线图等）应在 15 秒内完成计算并准确呈现。 |

## 4.2 安全需求 

系统需构建严密的安全防御体系，保障政务数据的机密性与完整性。

| ID | 需求名称 | 需求描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **NFR-201** | **三级权限隔离 (RBAC)** | 系统必须建立严密的基于角色的访问控制模型，严禁跨权限范围的垂直或水平越权访问。 | 经过渗透测试，企业级用户严禁访问管理端页面，市级用户严禁跨区域查看非辖区企业数据，权限覆盖率达到 100%。 |
| **NFR-202** | **全生命周期审计留痕** | 对所有涉及数据状态变更的操作（如修改、删除、审核退回）实施全过程记录，确保行政审批的可追溯性。 | 审计日志必须完整记录操作人、IP、精确时间戳及修改前后的关键数值对比，且日志具备只读属性。 |
| **NFR-203** | **敏感数据存储加密** | 针对系统涉及的企业核心信息及用户鉴权凭证，采用行业标准加密算法进行深度脱敏保护。 | 数据库中用户密码必须采用 salt 加盐的哈希算法（如 SHA-256）存储。关键业务数据传输需通过 HTTPS 加密。 |

## 4.3 可用性需求

系统需兼顾不同层级用户的操作习惯与技术能力，提供简洁、直观、易上手的交互体验，降低培训成本与误操作率。

| ID | 需求名称 | 需求描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **NFR-301** | **界面友好性与一致性** | 系统界面风格应统一、简洁，符合政务系统视觉规范，功能模块布局清晰，操作路径合理。 | 经用户测试，90% 以上的首次使用者可在无培训情况下完成“数据填报”核心操作。 |
| **NFR-302** | **输入校验与错误提示** | 系统需对关键字段（如组织机构代码、联系电话、邮箱等）进行实时格式校验，并提供明确的错误提示。 | 测试中，输入错误格式数据时，系统应弹出红色提示框，并明确说明错误字段及修正建议。 |
| **NFR-303** | **多浏览器兼容性** | 系统需兼容主流浏览器（Chrome、Edge、Firefox），确保在不同终端下功能一致、显示正常。 | 在以上浏览器的最新版本中执行核心业务流程，页面布局无错乱，功能无失效。 |
| **NFR-304** | **数据可视化支持** | 系统需支持以图表形式（如饼图、折线图）直观展示就业数据趋势，并提供 Excel 导出功能。 | 在省级用户查询全省数据时，系统应自动生成可视化图表，并提供一键导出功能，导出数据完整无误。 |

## 4.4 可靠性需求

系统需具备高可用性与容错能力，确保在全省企业集中上报期内稳定运行，保障数据的一致性与完整性。

| ID | 需求名称 | 需求描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **NFR-401** | **系统可用性保障** | 系统需提供 7×24 小时连续服务，年可用率不低于 99.9%，年非计划停机时间不超过 8 小时。 | 监控系统记录的年可用性指标 ≥ 99.9%，非计划停机累计时间 ≤ 8 小时。 |
| **NFR-402** | **事务一致性与完整性** | 系统需通过数据库事务机制，确保数据在填报、审核、退回等操作中的原子性与一致性。 | 在并发测试中，数据提交失败或异常中断时，数据库状态应回滚至操作前状态，无数据丢失或重复。 |
| **NFR-403** | **实时监控与自动预警** | 系统需内置监控模块，实时采集 CPU、内存、磁盘 I/O 等关键指标，并在资源超限时触发预警。 | 当 CPU 使用率连续 3 分钟超过 85% 时，系统应通过邮件或短信通知运维人员。 |
| **NFR-404** | **故障恢复能力** | 系统应具备自动或手动的故障恢复机制，确保在硬件或网络故障后能快速恢复服务。 | 在模拟故障测试中，系统应在 30 分钟内恢复服务，数据不丢失。 |

## 4.5 可维护性需求

系统需具备良好的可维护性，便于后续功能扩展、问题排查与系统升级。

| ID | 需求名称 | 需求描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **NFR-501** | **模块化与可扩展性** | 系统应采用分层架构设计，业务逻辑与数据访问分离，支持后续模块的平滑扩展。 | 在新增业务模块时，无需修改现有核心代码，仅需新增模块并配置路由即可上线。 |
| **NFR-502** | **日志分级与可读性** | 系统日志需按级别（INFO、WARN、ERROR）分类输出，便于问题追踪与性能分析。 | 运维人员可根据日志快速定位错误发生的时间、模块及异常堆栈信息。 |
| **NFR-503** | **配置管理能力** | 系统关键参数（如上报时间窗口、并发阈值、预警阈值等）应支持动态配置与热加载。 | 修改配置后无需重启服务即可生效，配置项变更记录可查。 |
| **NFR-504** | **自动化备份与恢复机制** | 系统需支持定期自动备份数据库与关键文件，并提供一键恢复功能。 | 备份任务每日执行，恢复测试中可在 1 小时内恢复至最近备份点。 |
